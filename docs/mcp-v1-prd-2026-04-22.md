# Musashi MCP V1 PRD

Last updated: 2026-04-22

## Purpose

`musashi-mcp` should become the first clean agent-facing read surface for Musashi.

V1 should **not** expose raw database access.

V1 should expose a small set of high-signal market tools backed by Musashi data.

## Product goal

Let an LLM user ask simple market questions like:

- what does Kalshi say about X?
- show me this market
- how has this market moved recently?
- how many similar markets have resolved?

V1 is **market-level first**, not full event intelligence yet.

## Why this scope

The truth layer is real enough for market-level tools.
The event layer is not mature enough yet to make `get_event_intelligence` the flagship public primitive.

So V1 should ship a narrow, reliable surface first.

## Non-goals for V1

Do not build:

- raw SQL / Supabase browsing tools
- open-ended table access
- external signal tools (X/news)
- full event reasoning as a stable public primitive
- trading / execution tools
- write operations

## Data dependency

Source of truth should come from Musashi read services backed by:

- `markets`
- `market_snapshots`
- `market_resolutions`

MCP clients should not access Supabase directly.

## V1 tool set

### 1. `search_markets`

Purpose:
- find relevant market candidates from a user query

Inputs:
- `query: string` required
- `limit?: number` default `10`, max `25`
- `category?: string`
- `status?: "open" | "closed" | "resolved"`

Output:
- list of markets with concise fields:
  - `id`
  - `platform`
  - `platform_id`
  - `title`
  - `category`
  - `status`
  - `yes_price`
  - `no_price`
  - `closes_at`
  - `resolved`

Success criteria:
- useful for “find the right market first”

### 2. `get_market`

Purpose:
- fetch one market with its current state

Inputs:
- one of:
  - `market_id`
  - `platform_id`

Output:
- single market object:
  - `id`
  - `platform`
  - `platform_id`
  - `title`
  - `description`
  - `category`
  - `status`
  - `yes_price`
  - `no_price`
  - `volume_24h`
  - `open_interest`
  - `liquidity`
  - `spread`
  - `closes_at`
  - `settles_at`
  - `resolved`
  - `resolution`
  - `resolved_at`
  - `source_missing_at`

Success criteria:
- useful as the canonical “current market state” read

### 3. `get_market_history`

Purpose:
- show how a market moved over time

Inputs:
- one of:
  - `market_id`
  - `platform_id`
- `window?: "24h" | "7d" | "30d" | "all"` default `7d`
- `limit?: number` default `200`, max `1000`

Output:
- market identity block
- ordered list of snapshot points:
  - `snapshot_time`
  - `yes_price`
  - `no_price`
  - `volume_24h`
  - `open_interest`
  - `liquidity`
  - `spread`

Success criteria:
- lets an agent answer “how did this move?”

### 4. `get_market_resolution_context`

Purpose:
- give the simplest trust/reliability context available before event-layer maturity

Inputs:
- one of:
  - `market_id`
  - `platform_id`

Output:
- current market identity
- scoped resolution context:
  - `market_resolved: boolean`
  - `market_resolution: "YES" | "NO" | null`
  - `market_resolved_at: string | null`
  - `category_resolution_count: number | null`
  - `similar_market_resolution_count: number | null`
  - `notes`

Important:
- if scoped trust context cannot be computed honestly, return `null`
- do not fake confidence

Success criteria:
- enough context for “what do we know historically around this market?”

## Optional V1.5 tool

### `get_event_intelligence`

Status:
- experimental only

Only expose this after Event Layer v1 is merged and the output is honest.

If included before then:
- clearly label as experimental
- return conservative/null fields where data quality is weak

## Tool design rules

1. All tools are **read-only**
2. All tools should return **clean structured JSON**
3. No raw Supabase errors should leak to end users
4. If data is insufficient, prefer `null` over fabricated output
5. IDs and fields should be stable across calls where possible

## Error semantics

Every tool should distinguish:

- `not_found`
- `invalid_input`
- `upstream_unavailable`
- `internal_error`

Shape:

```json
{
  "ok": false,
  "error": {
    "type": "not_found",
    "message": "Market not found."
  }
}
```

Successful responses should use:

```json
{
  "ok": true,
  "data": { ... }
}
```

## Technical shape

Recommended structure inside `musashi-mcp`:

- `src/tools/search-markets.ts`
- `src/tools/get-market.ts`
- `src/tools/get-market-history.ts`
- `src/tools/get-market-resolution-context.ts`
- `src/clients/musashi-api.ts`
- `src/schemas/*.ts`

The MCP server should stay thin:
- validate inputs
- call Musashi read client
- normalize outputs
- return stable MCP tool results

## API dependency expectations

If current `musashi-api` does not provide exactly these read endpoints yet, do **not** push query complexity into MCP ad hoc.

Instead:
- document the missing read endpoints
- either add them cleanly in `musashi-api`
- or create one explicit Musashi read layer that MCP depends on

## Acceptance criteria

V1 is successful when:

1. a user can search for markets reliably
2. a user can fetch one market cleanly
3. a user can inspect recent history cleanly
4. a user can get honest resolution/trust context
5. the server does not expose raw DB concepts
6. the JSON shapes are stable and documented

## What “done” looks like

Example flow:

1. user asks: “What does Kalshi say about Fed cuts?”
2. `search_markets` returns candidates
3. user or agent selects one
4. `get_market` returns current state
5. `get_market_history` shows recent movement
6. `get_market_resolution_context` gives basic historical trust context

That is enough for the first Musashi MCP release to be useful.

## Short summary

MCP V1 should be:

**agents talking to Musashi’s clean market tools**

not:

**agents poking around raw Supabase or half-finished event logic**
